## 1. Executive Summary

Civic Watch is a Next.js 16 application for hyperlocal community issue reporting with AI-powered image categorization via Google Gemini. The core flow (upload photo → AI analysis → pin on map → community verification) is functional in a basic sense, but the project suffers from **several broken features**, **widespread THEME-GUIDE violations**, **no real-time functionality**, **zero Google AI Studio integration** (mandatory per rules), and **no automated tests**. A judge evaluating this submission would likely score it around **~39/100** based on the official evaluation matrix.

## 2. CRITICAL — Broken Features

### 2.1 Leaderboard / Profile: `verifiedBy` query is broken

**File:** `src/app/profile/page.tsx:48`

```ts
.ilike('verifiedBy', `%${currentUser.id}%`)
```

`verifiedBy` is defined as `TEXT[]` (PostgreSQL array) in `supabase_setup.sql:13`. The `.ilike()` operator does **not** work on array columns. This query silently returns zero results. The "Tracked Issues" tab on every user profile is effectively dead code.

**Fix:** Use Supabase's `.contains()` operator instead:
```ts
.contains('verifiedBy', [currentUser.id])
```

### 2.2 `fix_db.sql` is destructive

**File:** `fix_db.sql:1`

```sql
ALTER TABLE public.reports ADD COLUMN id UUID DEFAULT gen_random_uuid() PRIMARY KEY;
```

The `reports` table **already** has an `id UUID DEFAULT gen_random_uuid() PRIMARY KEY` column defined in `supabase_setup.sql:3`. Running this script will error out with "column `id` already exists". This file should be deleted.

### 2.3 Memory leak — `URL.createObjectURL` never revoked

**Files:**
- `src/app/report/page.tsx:79,83`
- `src/components/ResolveModal.tsx:52,55`

`URL.createObjectURL()` is called to generate preview URLs, but `URL.revokeObjectURL()` is never called in the cleanup. In a long session with multiple upload attempts, memory usage grows unbounded.

### 2.4 Duplicated `compressImage` function

**Files:** `src/app/report/page.tsx:31-71` and `src/components/ResolveModal.tsx:14-44`

The exact same 40-line image compression function is copy-pasted in two components. Should be extracted to `src/lib/image.ts`.

### 2.5 Auth session setup duplicated 4 times

**Files:**
- `src/components/AuthNav.tsx:12-22`
- `src/components/LiveMap.tsx:50-58`
- `src/app/report/page.tsx:19-29`
- `src/app/issue/[id]/page.tsx:23-24`

Every client component that needs the current user repeats the same `supabase.auth.getSession()` + `onAuthStateChange` pattern. Should be a single `useAuth()` hook in `src/lib/hooks.ts`.

### 2.6 No error boundaries

If any client component throws a runtime error, the entire page crashes with a blank white screen. No `<ErrorBoundary>` wrapping any route.

## 4. Security Vulnerabilities

### 4.1 Row Level Security (RLS) completely disabled

**File:** `supabase_setup.sql:18,50`

```sql
ALTER TABLE public.reports DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.comments DISABLE ROW LEVEL SECURITY;
```

Any authenticated Supabase user can read, modify, or delete **any** report or comment. There is no authorization check anywhere in the application.

### 4.2 API routes have no authentication

**Files:**
- `src/app/api/analyze-image/route.ts`
- `src/app/api/verify-resolution/route.ts`

These endpoints accept POST requests from anyone. No session validation, no rate limiting, no file size validation. An attacker could:
- Send arbitrary files to consume Gemini API quota
- Repeatedly call the endpoint to exhaust the API key budget

### 4.3 Storage bucket allows unauthenticated uploads

**File:** `supabase_setup.sql:31-36`

```sql
CREATE POLICY "Allow public uploads to reports bucket" 
ON storage.objects FOR INSERT 
TO public 
WITH CHECK (bucket_id = 'reports');
```

The `TO public` clause means even unauthenticated visitors can upload arbitrary files to the Supabase storage bucket.

### 4.4 Potential XSS — no input sanitization

**Files:**
- `src/components/DiscussionBoard.tsx:76` — renders `{c.content}` directly
- `src/app/issue/[id]/page.tsx:93` — renders `report.userName` directly
- `src/app/profile/page.tsx:123` — renders `user.user_metadata?.full_name` directly
- `src/app/report/page.tsx:155` — inserts `userName` into DB without sanitization

A user could post a comment containing `<script>alert('xss')</script>` and it would execute for every viewer.

### 4.5 Live API key in `.env.local`

The Gemini API key (`GEMINI_API_KEY`) is live and readable. While `.env*` is in `.gitignore`, it is present on disk. If the deployment platform exposes environment variables or if the `.env.local` is ever served, the key is compromised.

## 8. Database & Schema Issues

### 8.1 Missing indexes

No indexes on frequently queried columns:
- `reports.userId` — queried in profile page, leaderboard
- `reports.status` — filtered on map
- `reports.severity` — filtered on map
- `reports.created_at` — sorted in leaderboard, profile
- `comments.reportId` — queried in every discussion board load

### 8.2 No foreign key constraints

`comments.reportId` is `TEXT NOT NULL` but has no FK reference to `reports.id`. Deleting a report leaves orphaned comments.

### 8.3 Array type complexity

`verifiedBy TEXT[]` is convenient for storage but:
- Cannot be indexed efficiently
- Makes queries awkward (`.contains()` vs `.ilike()` confusion)
- Cannot be enforced with FK constraints

A join table (`verifications`) would scale better and allow timestamps per verification.

### 8.4 `fix_db.sql` must be removed

As noted in §2.2, this file attempts to add a column that already exists and will error.
